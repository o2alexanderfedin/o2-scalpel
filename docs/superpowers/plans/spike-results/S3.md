# S3 - workspace/applyEdit reverse-request on command-typed code actions (post Stage 1A)

**Outcome:** B - all 9 resolved actions carry `edit:` directly; no command-typed action surfaced (minimal stub sufficient, re-verify S4/S5/S6)

**Evidence:**

- Code actions surfaced (lib.rs add-name + whole-fn ranges): 9
- Resolved actions (after `codeAction/resolve`): 9
- `command:`-typed actions (command set, edit empty): 0
- `edit:`-typed actions: 9
- executeCommand attempts: 0
- `workspace/applyEdit` reverse-requests observed: 0

**API audit (post Stage 1A T2/T6/T7/T8):**

- rust-analyzer returns deferred-resolution actions: top-level response carries only `{title, kind, data, group?}`; `command`/`edit` populate only after `codeAction/resolve`. Stage 1A code-action flow MUST resolve before classifying.
- `SolidLanguageServer.request_code_actions` / `.resolve_code_action` / `.execute_command` (T6/T7/T8) replace raw `srv.server.send_request(...)` calls. `execute_command` returns `(response, drained_apply_edits)` — drained list IS the applyEdit-payload list.
- T2 installs `_handle_workspace_apply_edit` by default; payloads captured into `_pending_apply_edits` (thread-safe) and drained via `pop_pending_apply_edits()` or the second tuple element of `execute_command`. Custom `on_request` registration + dispatcher-tee monkey-patch are no longer needed and have been removed from this spike.

**Decision:**

- A -> Stage 1A implements full applyEdit handler in solidlsp/lsp_protocol_handler/server.py (+80 LoC) delegating to WorkspaceEditApplier.
- B -> Stage 1A ships minimal `{applied: true, failureReason: null}` stub (~+20 LoC); reclaim ~50 LoC. Re-verify in S4/S5/S6 with richer fixtures (ssr, expandMacro, auto_import).
