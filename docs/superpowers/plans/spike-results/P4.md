**Wrapper-gap findings:**

- No `BasedpyrightServer(SolidLanguageServer)` adapter under `src/solidlsp/language_servers/`. `Language.PYTHON` resolves to PyrightServer (`src/solidlsp/ls_config.py:346`), so the existing `python_lsp_pylsp` fixture would actually boot vanilla pyright, not basedpyright. Test bypasses the wrapper with `_basedpyright_client.BasedpyrightClient` (parallel to `_pylsp_client.py` and `_ruff_client.py`). Stage 1E `PythonStrategy` should add a real adapter.
- basedpyright 1.39.3 serves diagnostics in PULL MODE ONLY: after `initialized` it `client/registerCapability` registers `textDocument/diagnostic` and emits ZERO `textDocument/publishDiagnostics`. The Stage 1E adapter must call `textDocument/diagnostic` explicitly after each `didOpen`/`didChange`/`didSave`, or the diagnostics_delta will always be empty.

**Decision:**

- A -> `RefactorResult.diagnostics_delta.severity_breakdown` exposes per-diagnostic `related_locations: list[Location]`. Applier and MCP serialization add ~+15 LoC for the field; UI/test consumers gain multi-file pointers for free. Empirically populated for `reportIncompatibleMethodOverride` (pointer to base-class method) and `reportCallIssue` (pointer to closest-matching overload). NOT populated for `reportArgumentType` on a same-file callee, so consumers must tolerate empty `related_locations`.
- B -> severity_breakdown stays minimal (title + range only); document gap and defer the related-locations field to v1.1 (when richer Python type-error chains actually need it).
- INDETERMINATE -> file basedpyright cold-start / pull-mode discovery as a Phase 0 finding; Stage 1E adapter must call `textDocument/diagnostic` (pull) instead of listening for `publishDiagnostics`.
