# v2.0 spec — MCP wire-name cleanup

**Date:** 2026-05-03
**Status:** DRAFT — pending user approval before plan-phase
**Milestone:** v2.0 (coordinated breaking change with deprecation window)
**Replaces:** none
**Related:** [`2026-04-29-lsp-feature-coverage-spec.md`](2026-04-29-lsp-feature-coverage-spec.md) (routing convention authority — keep `PREFERRED:` / `FALLBACK:` opener as routing signal)

---

## 1. Problem statement

Today's MCP tool wire names look like this:

```
mcp__plugin_o2-scalpel-rust_scalpel-rust__scalpel_capabilities_list
```

Per-segment decomposition:

| Segment | Source | Value | Carries |
|---|---|---|---|
| `mcp__plugin_` | Claude Code host (constant) | — | namespace prefix |
| `<plugin-name>` | `.claude-plugin/plugin.json` field `name` | `o2-scalpel-rust` | brand + language |
| `<server-name>` | `.mcp.json` `mcpServers` JSON-key | `scalpel-rust` | brand + language *(redundant)* |
| `<tool-name>` | `Scalpel*Tool` class name → `tools_base.py:178 get_name_from_cls()` | `scalpel_capabilities_list` | brand + verb *(brand-prefix redundant)* |

**Net redundancy:** "scalpel" appears 3×, "rust" (or `<lang>`) appears 2×. Each repeat costs LLM context tokens at every tool-call site, every tool-list prompt, every log line. With 46 tools × 52 plugins, the waste compounds.

## 2. Goals

1. **Each wire-name segment does unique work.** Plugin segment carries brand+language identity (the install-time anchor). Server segment carries the server's role (not its language). Tool segment carries the verb (not the brand).
2. **No collision risk introduced.** Existing per-plugin namespacing already disambiguates same-named server-keys across plugins; verify and use it.
3. **No breaking change for users without an alias-shim deprecation window.** Both old and new tool names + server-keys must be live for one full release cycle (v2.0 → v2.1).
4. **Routing convention preserved.** The `PREFERRED:` / `FALLBACK:` docstring opener (drift-CI-enforced) is the routing signal; the `scalpel_` name prefix is purely cosmetic and safe to drop.
5. **Marketplace + plugin-name unchanged.** `/plugin install o2-scalpel-rust@o2-scalpel` continues to work — install flows, on-disk install paths, and slash-command names stay stable.

## 3. Non-goals

- **Plugin name rename (`o2-scalpel-rust` → anything else).** Rejected: the marketplace + install command + on-disk dir + statusline glob + 411 doc occurrences + memorized user commands all key off this name. One-way break with HIGH user disruption for the same single-token reduction L1+L2 deliver.
- **Mega-plugin collapse (one plugin auto-detects language).** Rejected: kills the per-language curated facade-set model and the "install only what I use" knob. Architectural pivot, not a naming tweak.
- **`lean` → `lean4` generator-key alignment.** Out of scope (separate concern flagged in v1.14 follow-ups).
- **Shipping the 7 alternate adapters as plugins** (`cpp_ccls`, `csharp_omnisharp`, `php_phpactor`, `python_jedi`, `python_ty`, `ruby_solargraph`, `typescript_vts`). Out of scope; needs directory-naming-convention discussion.

## 4. Target wire format

```
mcp__plugin_o2-scalpel-rust_lsp__capabilities_list
            └── brand+lang ──┘ └role┘  └── verb ──┘
```

**Per-segment:**

| Segment | New value | Source change |
|---|---|---|
| `<plugin-name>` | `o2-scalpel-rust` | unchanged |
| `<server-name>` (JSON-key) | `lsp` (constant across all 52 plugins) | `plugin_generator.py:208` template |
| `<server-name>` (`--server-name` CLI arg) | `scalpel-rust` (per-plugin, **unchanged**) | `plugin_generator.py:217` — keeps Claude Code's args-uniqueness AND the existing dashboard `pgrep` pattern working |
| `<tool-name>` | `capabilities_list` (drop `scalpel_` prefix) | 46 `Scalpel*Tool` class renames + `tools_base.py:178` derivation unchanged |

**Result:** `mcp__plugin_o2-scalpel-rust_lsp__capabilities_list` — brand 1×, language 1×, role 1×, verb 1×, no redundancy.

## 5. The two changes

### L1 — Drop `scalpel_` prefix from tool names

**Mechanism:** rename 46 tool classes in the engine.

| File | Change |
|---|---|
| `vendor/serena/src/serena/tools/scalpel_facades.py` | 35 `class Scalpel<Verb>Tool` → `class <Verb>Tool` |
| `vendor/serena/src/serena/tools/scalpel_primitives.py` | 11 `class Scalpel<Verb>Tool` → `class <Verb>Tool` |
| `vendor/serena/src/serena/tools/__init__.py` | import list updated |
| Module filenames | Optional rename: `scalpel_facades.py` → `facades.py`, `scalpel_primitives.py` → `primitives.py`. Defer to keep blast radius scoped. |

**Catalog hash regen** (`vendor/serena/src/serena/refactoring/capabilities.py:131`): the `preferred_facade` field carries 11 literal `scalpel_*` strings. Rebaseline:

```sh
PATH="vendor/serena/.venv/bin:$PATH" .venv/bin/pytest \
  test/spikes/test_stage_1f_t4_baseline_round_trip.py --update-catalog-baseline
```

**Drift-CI** (`vendor/serena/test/serena/tools/test_docstring_convention.py:60`): currently filters by `cls.__name__.startswith("Scalpel")`. Update to track via marker mixin (`ToolMarkerScalpelFacade`) or module-of-origin filter (`cls.__module__.endswith(".scalpel_facades")` or `.facades`). The drift-CI itself stays — only the *filter expression* changes, not the assertion semantics.

**Routing safety:** `PREFERRED:` / `FALLBACK:` docstring opener is unaffected. Tool prefix was cosmetic; routing is docstring-driven.

### L2 — Server JSON-key `scalpel-<lang>` → `lsp`

**Mechanism:** one line in the generator.

| File | Change |
|---|---|
| `vendor/serena/src/serena/refactoring/plugin_generator.py:208` | `server_name = f"scalpel-{strategy.language}"` → `server_key = "lsp"` (separate from CLI arg) |
| `vendor/serena/src/serena/refactoring/plugin_generator.py:217` | `--server-name` CLI arg stays `f"scalpel-{strategy.language}"` (unchanged) |
| `vendor/serena/src/serena/refactoring/templates/mcp.json.tmpl` | Template uses two distinct values: `mcpServers` JSON-key = `lsp`, args[…] `--server-name` value = `scalpel-<lang>` |

**Why JSON-key and CLI arg now diverge:**
- The **JSON-key** is what Claude Code reads to build the wire namespace. Setting it to `lsp` shrinks the wire name by 11 chars per call.
- The **CLI arg `--server-name scalpel-<lang>`** must stay per-plugin-unique because Claude Code's plugin manager dedupes plugins with identical args arrays. The engine itself ignores this arg at runtime (`vendor/serena/src/serena/cli.py:301-309`, marked `hidden=True`).
- Keeping the CLI arg unchanged also means the existing dashboard `pgrep -f "--server-name scalpel-<lang>"` pattern in `commands/o2-scalpel-<lang>-dashboard.md` works without modification.
- The update-command `pgrep -f "--server-name scalpel-"` prefix-match in `update_command.md.tmpl:40` also works unchanged.

**Cross-plugin namespace safety:** Claude Code namespaces by plugin (`mcp__plugin_<plugin>_<server>__`). 52 plugins all using JSON-key `lsp` produce 52 distinct wire namespaces — no collision.

## 6. Migration plan — alias shims

### L1 alias shim

**Approach:** dual-register every renamed tool under both old and new names in `ToolRegistry`. The registry already keys by `tool_name` (a string), so emitting two registry entries per class is one loop.

**Pseudocode** (in tool registration site):
```python
for cls in scalpel_tool_classes:
    new_name = cls.get_name()           # e.g. "split_file"
    legacy_name = f"scalpel_{new_name}" # e.g. "scalpel_split_file"
    registry.register(new_name, cls)
    registry.register(legacy_name, cls)  # alias for v2.0 deprecation window
    # Add deprecation warning header in legacy entry's description:
    # "[DEPRECATED in v2.0; use `<new_name>`. Removed in v2.1.]"
```

**Description-field deprecation marker** for the legacy alias makes it observable in tool-list dumps and lets users (and Claude) discover the new name organically.

**Removal:** drop legacy aliases in v2.1.

### L2 alias shim

**Approach:** Claude Code reads the `.mcp.json` JSON-key once at plugin install and uses it permanently for wire-namespacing. There is no in-flight aliasing path at the engine level.

**Migration:** users `/plugin update o2-scalpel-<lang>@o2-scalpel` (or uninstall + reinstall) once. Claude Code re-reads the new `.mcp.json` and uses the new server key.

**During v2.0 deprecation window:** L1 aliases mean both `mcp__plugin_o2-scalpel-rust_lsp__capabilities_list` (new) and `mcp__plugin_o2-scalpel-rust_lsp__scalpel_capabilities_list` (L1-aliased legacy, new server name) work. Users who have NOT updated their plugin still see the v1.x name `mcp__plugin_o2-scalpel-rust_scalpel-rust__scalpel_capabilities_list` and their workflows keep working.

**v2.0 release notes** must explicitly call out: "Run `/plugin update` for full v2.0 ergonomics." Users who skip the update get the legacy wire format until they update.

### Combined deprecation timeline

| Release | What ships |
|---|---|
| **v2.0** | New tool names (no `scalpel_` prefix) live. Old tool names live as L1 aliases with `[DEPRECATED]` marker. New server JSON-key `lsp` live for users who `/plugin update`. |
| **v2.0.1 – v2.0.x** | No further breaking changes. Possible doc + CHANGELOG polish. |
| **v2.1** | L1 aliases removed. Legacy `scalpel_*` tool names return `MCP error: tool not found`. v2.0 release notes warned about this. |

## 9. Rollback plan

### If L1 surfaces a regression in v2.0

- **Symptom:** routing benchmark drops more than 5pp, or a key user workflow breaks.
- **Rollback:** revert the class-rename commits (single-file PR per side). The L1 alias shim was the new code path; old `Scalpel*Tool` class names are reachable via `git revert`. Catalog baseline regenerates automatically.

### If L2 surfaces a collision (unexpected)

- **Symptom:** users report `mcp__plugin_o2-scalpel-X_lsp__…` collides with a non-Scalpel MCP server they have installed.
- **Likelihood:** very low (Claude Code namespaces by plugin), but worth mentioning.
- **Rollback:** revert `plugin_generator.py:208` to the per-language `scalpel-<lang>` JSON-key.

### If users skip `/plugin update` and complain

- **Reality:** they keep working with v1.x wire names. Only the dashboard `pgrep` needs the new server name to discover the dashboard port — and that pattern keys off the CLI arg, which we kept unchanged. So even un-updated users see no breakage.
- **No rollback needed.** Just better release notes.

## 10. Open questions

1. **Should we also rename the source-file paths** (`scalpel_facades.py` → `facades.py`)? Pro: consistency with the prefix drop. Con: doubles the rename diff and breaks any historical git-blame links. **Recommendation: defer to v2.1 cleanup.**

2. **Should the L1 deprecation marker include a remove-in-version hint?** E.g. `[DEPRECATED in v2.0; removed in v2.1; use <new_name>]`. **Recommendation: yes** — gives users a concrete migration deadline.

3. **Do we need a `scalpel-mcp-status` MCP tool that prints "you're on v2.0; update via /o2-scalpel-update"?** Could help users self-diagnose if they're seeing the legacy wire format. **Recommendation: defer; the existing `scalpel_workspace_health` already reports engine version, which users can grep for.**

4. **Does the marketplace catalog hash include the server JSON-key?** If yes, the marketplace itself bumps version on this change. Verify in plan-phase.
